# deep_research辅助脚本

这个子目录存放的是几份**面向 Deep Research / 网页整理稿 / 半结构化 Markdown** 的后处理小脚本。

它们的定位不是完整文献管理流水线，而是：

- 先把 Markdown 里的引用编号、参考文献区、链接格式整理到一个更稳定的状态
- 再决定是否进入主目录下的 BibTeX / `bibkey` 工作流

如果你的目标是：

- 生成 `.bib`：优先看上一级目录的 `obsidian_bib_extractor.py`
- 回填 `[@citekey]`：优先看上一级目录的 `reference_bibkey_mapper.py` 和 `obsidian_cite_replacer.py`

## 脚本说明

### 1. `fix_citations.py`

**用途**

- 规范化参考文献行
- 把正文里形如“ 空格 + 数字 + 句号”的内联编号改成带链接的引用形式

**它会做什么**

- 识别参考文献区中类似：
  - `1. 标题 [url](url)`
  - `1. [标题](url)`
- 清理末尾的访问时间描述
- 统一改成：
  - `1. [标题](url)`
- 再把正文中的：
  - ` 1。`
- 替换成：
  - `[1](url)。`

**运行方式**

这个脚本会从当前工作目录开始，递归处理所有 `.md` 文件。

**示例**

```bash
cd "/path/to/target-dir"
python fix_citations.py
```

**注意**

- 它会直接覆盖原文件
- 建议先在副本目录测试，或先自行备份

---

### 2. `clean_inline_citations.py`

**用途**

- 扫描整篇 Markdown 中的普通链接
- 给正文中的链接统一分配数字引用编号
- 在文末重建一个 `# References` 区块

**适合场景**

- 正文中散落很多 `[标题](url)` 链接
- 想把这些链接合并成统一编号引用

**主要行为**

- 相同 URL 只分配一个编号
- 会清除文末已有的 `# References` 区块后重新生成
- 会去掉部分常见跟踪参数

**输入**

- 单个 Markdown 文件

**输出**

- 原文件被直接改写

**示例**

```bash
python clean_inline_citations.py "/path/to/file.md"
```

**注意**

- 这是“全文件重写型”脚本
- 如果原来的 `# References` 区块有手工补充内容，可能会被覆盖

---

### 3. `clean_reference_list.py`

**用途**

- 清洗文末参考文献区
- 合并重复链接
- 重排正文中的引用编号
- 重新生成更整齐的 `**参考文献：**` 列表

**适合场景**

- 文末已经有一份参考文献区，但格式杂乱
- 同一个 URL 在正文和参考文献中重复出现
- 想统一编号和展示格式

**主要行为**

- 识别 `**参考文献：**` 区块
- 如果没找到，也会尝试根据分隔线 `---` 推断参考文献区
- 去重 URL
- 保留标题、描述等文本信息
- 重建正文和参考文献的编号映射

**输入**

- 单个 Markdown 文件

**输出**

- 原文件被直接改写

**示例**

```bash
python clean_reference_list.py "/path/to/file.md"
```

**注意**

- 这个脚本对“参考文献区块格式”有一定假设
- 如果原文结构非常自由，最好先用样例文件试跑

---

### 4. `clean_references.py`

**用途**

- 重新解析引用区块
- 合并多个旧编号到新的唯一编号
- 在文末生成一个新的 `# References` 列表

**适合场景**

- 原文正文已经是 `[1](url)` 这类编号引用
- 但文末参考文献块重复较多、编号混乱

**主要行为**

- 尝试识别参考文献区开始位置
- 读取每个引用块中的旧编号和主 URL
- 建立“旧编号 → 新编号”的映射
- 重写正文中的引用编号
- 重建新的 `# References`

**输入**

- 单个 Markdown 文件

**输出**

- 原文件被直接改写

**示例**

```bash
python clean_references.py "/path/to/file.md"
```

**注意**

- 这个脚本对参考文献区的检测依赖启发式规则
- 如果找不到 `\n[1](` 这样的起始特征，可能无法处理

## 这 4 个脚本怎么选

### 情况 A：正文里都是普通超链接，文末没有稳定参考文献区

优先试：

- `clean_inline_citations.py`

### 情况 B：文末已有“参考文献”区，但很乱、重复多

优先试：

- `clean_reference_list.py`

### 情况 C：正文已经是 `[1](url)` 这种编号引用，但编号体系混乱

优先试：

- `clean_references.py`

### 情况 D：你只想把“参考文献标题 + 正文数字”做一次轻量修复

优先试：

- `fix_citations.py`

## 推荐使用顺序

建议不要一次把 4 个脚本串起来盲跑。

更稳妥的方式是：

1. 先复制一份 Markdown 样本
2. 选一个最贴近当前问题的脚本试跑
3. 检查输出是否符合预期
4. 再决定是否继续使用上一级目录的主流程脚本

通常建议：

1. 先做结构清理
2. 再做 BibTeX 抽取或 `[@citekey]` 回填

---

## 风险提示

这几个脚本的共同特点是：

- 都会**直接覆盖原文件**
- 都依赖一定的格式假设
- 更适合处理“批量相似格式”的研究稿，而不是完全自由排版的 Markdown

因此建议：

- 先在副本目录试跑
- 或先用 Git / 备份保存原稿
